# Market Pulse AI: Personal Investment Assistant Roadmap (v1)

## 1. Executive Summary

The objective of Market Pulse AI is to evolve into a highly personalized, auditable, and safe US equity investment assistant. Rather than functioning as a black-box trading bot where a Large Language Model (LLM) freely generates and executes trades, the system will operate as a **Factor-Driven, Agent-Assisted Multi-Modal Platform**. 

The LLM's role is strictly confined to interpreting unstructured data (news, SEC filings, earnings calls) and participating in debate (Swarm Debate). Final recommendations are generated by a structured scoring engine, backed by a persistent memory of past mistakes and benchmark-relative out-of-sample backtesting. Interactive Broker (IBKR) integration will focus heavily on read-only synchronization, portfolio attribution, and trade reviewing, with execution firmly behind a human-in-the-loop approval gate.

## 2. What the Current Project Does & Architecture Overview

**Current State Analysis:**
- **Data Collectors (Breadth is high):** `server.mjs` integrates Longbridge, AkShare, SEC EDGAR, Reddit/StockTwits (social heat), and options chains.
- **LLM Workflows:** Uses a primitive "Agent Debate" which is currently a set of hardcoded Javascript rules (`buildAgentDebate`), not genuine LLM-driven agents.
- **IBKR Integration:** There are existing configurations and bridges (`IBKR_FLEX_TOKEN`, `IBKR_CP_BASE_URL`, `scripts/ibkr_gateway_bridge.py`) for market data and some trading functionality, but it lacks a mature risk-managed abstraction layer.
- **Storage Model:** Relies heavily on a monolithic `store.json` which is periodically mirrored to SQLite (`scripts/sqlite_store_sync.py`).
- **UI & Strategies:** Has basic HTML/JS dashboards (`app.js`, `index.html`) displaying candidate lists and backtest outputs, governed by `strategies/all_stock_agent_skill.md`.

## 4. Recommended Architectures

### 4.1 Product Architecture
1. **Market Radar Module:** Daily macro regime, sector heatmaps, and top catalysts.
2. **Deep-Dive Dashboard:** Single stock 360° view (Business, Supply Chain, Options flow, Financials).
3. **Recommendation Pipeline:** Generates Daily Actionable Candidates vs. Research Ideas.
4. **Post-Trade Review & Portfolio Analyst:** IBKR sync, PnL attribution, and behavioral trade reviews.

### 4.2 Data Architecture
- **Deprecate `store.json` Monolith:** Transition primary read/write operations to SQLite/PostgreSQL to support relational queries (`recommendation_decisions`, `factor_snapshots`, `trade_reviews`).
- **Data Quality Audit Layer:** Every data fetch must be stamped with a `DataQualityScore`. If quality drops below 60, the asset is automatically vetoed for trading.

### 4.3 LLM / Agent Architecture
- **Swarm Debate Harness:** Move away from hardcoded logic. Implement a Map-Reduce agent harness.
  - *Map:* Spin up `Fundamental`, `Technical`, `News`, and `Risk` LLM instances concurrently.
  - *Reduce:* A `CIO (Chief Investment Officer)` LLM evaluates the expert inputs.
- **Structured Output:** All LLM outputs must conform to strict JSON schemas to guarantee programmatic intervention.

### 4.4 Recommendation-Engine Architecture
- **Factor Snapshot:** Raw data is converted into normalized z-scores grouped by industry and market cap.
- **Scoring Formula:** `ActionScore = (AlphaScore * DataQualityMultiplier * RegimeMultiplier) + PortfolioFit - RiskPenalty`.
- **Hard Veto:** The system *cannot* buy if there is no trend data, if IV crush risk is high, or if the Risk Agent issues a veto.

### 4.5 IBKR Architecture (Safe Execution)
- **Phase 1 (Read-Only):** Ingest IBKR Flex Queries and Client Portal (CP) APIs strictly for position sync and execution history.
- **Phase 2 (Paper):** Paper trading against real-time IBKR market data.
- **Phase 3 (Live with Gate):** Draft orders are generated via API but require the user to explicitly click "Submit" on the frontend. *Fully autonomous live trading is explicitly forbidden.*

## 5. Proposed Database Schemas (SQLite)

- `factor_snapshots`: `id`, `ticker`, `as_of`, `momentum_score`, `valuation_score`, `data_quality`.
- `recommendation_decisions`: `decision_id`, `ticker`, `action_score`, `thesis`, `invalidation_conditions`, `benchmark_basket`, `created_at`.
- `ibkr_trade_history`: `trade_id`, `ticker`, `action`, `fill_price`, `quantity`, `commission`, `timestamp`.
- `post_trade_reviews`: `review_id`, `trade_id`, `behavior_flag` (e.g., FOMO, Late Stop-loss), `lessons_learned`.

## 6. Challenging Weak Assumptions

1. **Should LLM generate direct trade recommendations?**
   > **NO.** LLMs hallucinate and lack strict mathematical rigor. They should only output discrete evidence (e.g., "Earnings revision is positive"). The final Buy/Sell signal MUST be computed by a deterministic formula (`ActionScore`) that we can backtest.
2. **Do daily recommendations create overtrading risk?**
   > **YES.** To mitigate this, introduce a `Portfolio Fit / Position Sizing` constraint. If the portfolio is already 80% invested, the threshold for a new "Buy" recommendation must scale dynamically (e.g., requiring an ActionScore > 90 instead of 75).
3. **Are social/news hotspots predictive or noisy?**
   > Mostly noisy. Social heat should only serve as a *filter* (identifying liquidity) or as an *exit signal* (overcrowded trade), never as a primary alpha driver for entry.
4. **Is options-flow data reliable enough?**
   > Retail options data is notoriously noisy. It should only be used as a *risk gate* (e.g., extremely high IV percentile signals an upcoming catalyst/crush) rather than a direct buy signal.
5. **Is "Self-Improvement" safe?**
   > Uncontrolled self-modification leads to overfitting and strategy drift. The system will only "learn" by updating a text-based `Persistent Memory` database (e.g., "Avoid retail stocks during high inflation") and through strictly constrained weight tuning (max ±2% shift per month based on rank IC).

## 7. Prioritized Roadmap & Implementation Tasks

### P0 — Foundation (Months 1-2)
- **Task 1: Codebase Storage Migration.** Migrate `store.json` writes to SQLite for `recommendation_decisions` and `factor_snapshots`.
- **Task 2: Data-Quality Audit Layer.** Implement `calculateDataQuality()` for all incoming Longbridge/SEC feeds.
- **Task 3: Unified Factor Snapshot.** Build `lib/recommender/factors.mjs` to convert raw data into standard z-scores.
- **Task 4: Safe IBKR Read-Only Sync.** Ensure `ibkr_gateway_bridge.py` only pulls account balances and historical Flex Query logs.

### P1 — Investment Assistant Core (Months 3-4)
- **Task 1: Single-Stock Research Page.** Upgrade `public/app.js` UI to show Peer Comparison, Supply Chain (Upstream/Downstream), and Factor Spider Charts.
- **Task 2: Swarm Debate Harness.** Implement `runAgentHarness()` to spawn Fundamental, Technical, and Risk LLMs.
- **Task 3: Post-Trade Review Engine.** Create an LLM workflow that takes IBKR `ibkr_trade_history` and outputs a behavioral review (e.g., "You held this loser 15% past your defined stop-loss").

### P2 — Learning Loop (Months 5-6)
- **Task 1: Benchmark-Relative Outcomes.** Update the backtester to score against dynamic `benchmark_basket` (e.g., SMH for semi stocks) instead of just SPY.
- **Task 2: Persistent Memory.** Build an endpoint to inject failed thesis summaries into the context window for future recommendations.
- **Task 3: Strategy Versioning.** Track precision@10, excess return, and hit rate per `modelVersion`.

### P3 — Advanced Assistant (Months 7+)
- **Task 1: Portfolio-Level Risk Allocation.** Assess overall portfolio concentration (e.g., "You are 60% exposed to semiconductors").
- **Task 2: Live-Order Draft Workflow.** Generate IBKR Draft Orders that sit in a "Pending Approval" UI queue.
- **Task 3: Multi-modal Review Reports.** Generate PDF/HTML reports with overlaid entry/exit markers on technical charts using LLM vision capabilities.

## 8. Open Questions and Technical Risks
- **IBKR Gateway Stability:** The current Java/Python IBKR Gateway bridge is notoriously brittle and prone to disconnects. A strict reconnect/keep-alive daemon is required.
- **LLM API Costs & Latency:** Spawning 4-5 LLM calls per stock daily will quickly hit API rate limits and increase costs. Caching and localized LLM deployment (e.g., Llama-3 via Ollama) for non-critical agents should be evaluated.
- **Look-Ahead Bias in Backtesting:** Relying on current Longbridge API data for past dates often includes restated financials. We must ensure we use strictly Point-In-Time (PIT) datasets if available.
